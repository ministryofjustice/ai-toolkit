# Naming Things

- Make names self-descriptive and consistent; reuse the same name for the same concept (e.g. a GitHub repository name matches its hostname).
- Avoid puns, jokey or cute names, uncommon acronyms, version numbers, or brand names that obscure purpose.
- Avoid meaningless names such as `obj`, `result`, or `foo`.
- Use single-letter names only for well-known maths (e.g. `e = mc²`).
- Prefer well-named code over comments; good names reduce the need for explanation.
- Prefer clear service names, for example: Peoplefinder, Send Money to a Prisoner, Help with Fees.
- Avoid ambiguous names and internal acronyms (e.g. CCCD, PVB2, DOMIS).
